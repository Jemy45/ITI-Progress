Note:
Flannel is a CNI (Container Network Interface). Think of it as a massive, invisible virtual switch that connects all your servers together so your Pods can talk to each other.
Because we are using k3s so we dont need to handle this part if we using kubeadm as previous w need to apply it manually it opened UDP port 8472 on the firewall 

If a Pod on Server A tries to talk to a Pod on Server B using that fake 10.42 IP address, your physical router will be completely confused and drop the connection, because those IPs don't exist on your real network.
Flannel's Job:
It intercepts the traffic from the Pod.
It wraps that traffic up in a digital envelope (called encapsulation).
It writes the real physical IP address of Server B on the outside of the envelope.
It sends it across your real network.
Flannel on Server B catches it, opens the envelope, and hands the original message to the correct Pod.

Notes: dry-run have 3 cases 
--dry-run=none => this is the default, Kubernetes processes your command, talks to the master server, saves the object into the database, and actually builds the Pods.

--dry-run=client => (The Local Draft) the kubectl tool on your computer generates the YAML configuration locally. It does not send any network traffic to the Kubernetes Master Node.(Best way to get draft for objects)

-dry-run=server (The Pre-Flight Check)your computer actually sends the request to the Kubernetes Master Node. The master server runs all its security checks, verifies the syntax, and ensures you have enough permissions. But it stops right before saving the object to its internal database. It replies back to you saying, "This is valid and it will work when you are ready."

notes:
The StripPrefix Middleware Example:
When you ask for world.universe.mine/africa, the Pod crashes because it doesn't have an /africa folder; it only understands the root path (/).

The request comes in as /africa.

The Middleware intercepts it.

The Middleware "strips" (deletes) the word africa from the request.

The Middleware hands a clean / to the Pod.

The Pod understands /, finds the webpage, and sends it back to you.

-----------------------------------------------------
Notes:
When you open your browser and go to http://world.universe.mine/africa/:

Your computer uses /etc/hosts to send the traffic to your Master Node.

The Ingress Controller on the Master Node catches it and hands it to the Africa Service.

The Africa Service says, "I have two pods. One is right here on the Master, and one is over on the Worker."

It picks one at random and forwards the traffic. If it picks the pod on the Worker, the Master node silently routes the traffic over the internal cluster network to the Worker node in milliseconds.
So it doesn't matter if i add worker ip or master ip inside /etc/hosts

Explain the line of the rule in one sentence:
If a user visits world.universe.mine and goes to the /europe/ page, grab that traffic and hand it over to the internal service named europe on port 8888.
